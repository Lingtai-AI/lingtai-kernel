# Dedup (Duplicate Message Gate)

> **Protocol:** mail-protocol / send / dedup

---

## What

The dedup gate prevents runaway auto-reply loops where two agents endlessly reply to each other with identical payloads. It sits in the `EmailManager` wrapper (capability layer), **before** any Mailman thread is spawned — once a message passes the gate, delivery proceeds without further dedup checks.

The mechanism is simple: an in-memory counter tracks consecutive identical messages per recipient address. When the count reaches the threshold, the send is blocked with a warning.

---

## Contract

### Data structure

```python
# EmailManager.__init__
self._last_sent: dict[str, tuple[str, int]] = {}
self._dup_free_passes = 2  # allow this many identical sends
```

- **Key:** recipient address (`str`)
- **Value:** `(message_text, consecutive_count)` — count of how many times the same `message_text` has been sent to this address in rapid succession.

### Gate logic

```
for each addr in (to + cc + bcc):
    if _last_sent[addr] exists
       AND message_text matches previous text
       AND count >= _dup_free_passes (2):
        → BLOCK (return {"status": "blocked", "warning": "..."})
```

If any one recipient is blocked, the **entire send** is aborted — no recipients receive the message.

### After successful delivery

For each recipient address:
- If the address already has an entry and the text matches → increment count.
- If the address has no entry or the text differs → reset to `(text, 1)`.

### Bypass: scheduled sends

When `args.get("_schedule")` is truthy, the gate is skipped entirely. Scheduled messages are exempt because they represent intentional recurring communication, not accidental loops.

### Lifecycle

- **In-memory only** — `_last_sent` is a Python dict, not persisted to disk.
- **Agent-lifetime scope** — cleared on agent restart (fresh `EmailManager` instance).
- **No cross-agent awareness** — each agent's gate only tracks its own sends.

## Architectural note: no UUID linkage across sent/inbox

The wrapper's `sent/` record and the Mailman-produced `inbox/` record use **independent UUIDs** with no cross-reference field. The three UUIDs in a single send (outbox → inbox → sent) are unlinked:

- **outbox UUID** — generated by `_persist_to_outbox()`, deleted after Mailman completes (transient).
- **inbox UUID** — generated by `_persist_to_inbox()` or `FilesystemMailService.send()` (permanent).
- **sent UUID** — generated by the wrapper (permanent).

The dedup gate operates entirely on **content matching** (`message_text` comparison), not on UUID tracking, so this unlinking has no impact on dedup correctness. However, this means there is no mechanism for "delivery confirmation" — you cannot query whether a specific `sent/` record was successfully received without content-based correlation.
