# lingtai.mcp_servers

Curated and built-in MCP server package implementations shipped inside the `lingtai` Python distribution. They are launched by catalog/script entry points such as `python -m lingtai.mcp_servers.<name>` and expose real addon tools (IMAP, Telegram, Feishu, WeChat, WhatsApp, Cloud Mail), daemon lifecycle tools, plus bundled progressive-disclosure manuals.

## Components

| File / folder | Role |
|---|---|
| `_skill.py` | Shared bundled-skill helper: re-exports the kernel-owned `split_frontmatter` from `lingtai.kernel._frontmatter` (one impl shared with the prompt-section catalog; kernel never imports the wrapper), `load_skill()` loads package `SKILL.md`, `manual_action_description()` injects frontmatter into the schema, and `manual_payload()` returns the manual body + absolute path without sidecar lists (`_skill.py:36-79`). |
| `_identity.py` | Shared public-identity envelope/path/write helper for curated messaging MCPs: builds the `lingtai.mcp.identity.v1` document, computes `system/mcp_identities/<name>.json`, and performs the newline-terminated atomic JSON write. Provider-specific account fields and redaction stay in each provider. |
| `daemon_common/` | Built-in daemon lifecycle MCP. `daemon_common/server.py:1-151` exposes `finish(status, summary?, reason?, artifacts?)`, validates the call, and atomically writes the internal per-run `daemon_completion.json` file named by `LINGTAI_DAEMON_COMPLETION_FILE`; daemon runners validate that file before allowing success. |
| `telegram/`, `imap/`, `feishu/`, `wechat/`, `whatsapp/`, `cloud_mail/` | Curated messaging MCPs. TelegramManager requires an injected `NotificationStorePort`; `telegram/server.py` constructs one POSIX adapter, and handled-mirror policy runs against the current payload in one compare-update so newer mirrors survive (`src/lingtai/mcp_servers/telegram/manager.py:415-425`, `src/lingtai/mcp_servers/telegram/manager.py:1239-1281`, `src/lingtai/mcp_servers/telegram/server.py:655-663`). The external LICC path/envelope and persistent-message lanes remain unchanged. |
| Per-package `SKILL.md` | The human/agent-facing bundled manual. If a manual has sidecars, the sidecar inventory and relative paths live in this markdown, not in the tool payload. |
| `pyproject.toml` package-data entries | Ships every curated MCP `SKILL.md`; `reference/**/*` and `assets/**/*` are also packaged for future sidecar files (`pyproject.toml:81-86`). |

## Connections

- Catalog/script launchers (`pyproject.toml:43-49`) start these servers as subprocess MCPs; agents activate them through the generic MCP capability (`src/lingtai/tools/mcp/ANATOMY.md`).
- Manager schemas include `manual` in each action enum and use `_skill.manual_action_description()` to advertise the bundled skill without loading the full body into the resident schema.
- Tests pin the manual contract, package-data sidecar support, and Telegram parity in `tests/test_mcp_skill_manuals.py` and `tests/test_telegram_rich_formatting.py`.

## Composition

Parent: `src/lingtai/` wrapper package (`src/lingtai/ANATOMY.md`). Sibling wrapper areas include `agent.py`, `core/`, `services/`, and `intrinsic_skills/`. Curated MCPs are independent subprocess packages, not intrinsic capabilities.

## State

The package itself is mostly code + packaged manuals. Runtime state is per-agent and server-specific: e.g. message caches, contacts, inbox replay guards, credential-derived identities, or daemon completion sentinels live under the agent workdir/run dir or `.secrets/`, not in `src/lingtai/mcp_servers/`. The shared manual and identity helpers have no persistent state of their own. Telegram's per-account non-secret `state.json` (under `<workdir>/telegram/<alias>/`) additionally persists the resident Task Card id per chat under a `task_cards` map (`{chat_id: compound_id}`), loaded backward-compatibly and normalized against malformed values by `TelegramAccount._normalize_task_cards`, so the one-card-per-account+chat singleton survives a refresh and the next card can still delete the prior one. The same account additionally owns an **ephemeral, non-persisted** per-chat high-water mark of the latest observed Telegram message id (`_last_message_ids`, `telegram/account.py:190`), bumped by every inbound plain message the poll loop sees (`_process_update`) and every outbound `send_message`/`send_photo`/`send_document`, but never by an edit or delete; `_note_chat_message_id`/`get_last_message_id` (`telegram/account.py:1208,1231`) record and expose it, guarded by the same `_state_lock` as `task_cards`. Because Telegram message ids are monotonic per chat this mark is "the chat's last message id"; the manager reads it to keep the resident Task Card residing as the last message (Jason #5272/#5273/#5275). It is deliberately not persisted, so a refresh reports `None` (unknown) until the next observed message, and the manager treats an unknown latest-message mark conservatively by editing in place; malformed or cross-bound resident ids fail loud before transport, and neither condition authorizes deletion. Separately, `TelegramService` owns one current-agent delivery boolean in `<workdir>/telegram/taskcard.json`, loaded with a compatibility default of `True` and written through the kernel atomic JSON primitive with fsync; bound getter/setter callbacks let every configured account's local `/taskcard` command share that one value (`telegram/service.py`, `telegram/account.py`).

## Notes

- **Notification contract:** curated messaging MCPs that change structured notification metadata (`recent_messages`, `latest_incoming`, `referenced_messages`, stable IDs, routing hooks, or preview/body placement) must check `src/lingtai/services/LICC_NOTIFICATION_CONTRACT.md` in the same change.
- **Manual sidecar minimal contract:** `action="manual"` returns the main `SKILL.md` body, parsed metadata, and the main `SKILL.md` absolute `path` only. Concrete `assets/` and `reference/` lists MUST NOT be returned as structured tool fields; `SKILL.md` is the single source of truth for what sidecars exist and how to follow their relative paths.
- **Packaging discipline:** when adding manual sidecars, put their relative paths in `SKILL.md` and keep the package-data globs for `reference/**/*` / `assets/**/*` so wheels contain them (`pyproject.toml:81-86`).
- **Telegram private reverse-channel tool:** `telegram/server.py` `list_tools` advertises only the public `telegram` tool (validated against `SCHEMA` by the mcp library's default `validate_input=True`). Its `build_server._call_tool` also accepts one **unlisted** private tool name, `_PRIVATE_TASK_CARD_TOOL = "_lingtai_telegram_task_card"` (`telegram/server.py:64`), used solely by the kernel to project the live Task Card; being unlisted, it skips public-schema validation yet still reaches the handler, which forces `action="_task_card_update"` before `manager.handle` so the hidden route cannot invoke any public action. The mechanism, kernel caller (`_TASK_CARD_TOOL`), and regression tests are described in `src/lingtai/tools/mcp/ANATOMY.md`.
- **Resident Task Card singleton, one per account+chat; create is update-first and resides as the last message (Jason #6665/#6667, #6894/#6899, #5272/#5273/#5275).** `_handle_task_card_update` (`telegram/manager.py:1628`) dispatches create/update/finalize. Because the kernel's automatic task-card context is turn/request-local, every new BaseAgent tool batch/turn re-issues `create`; `_task_card_create` (`telegram/manager.py:1669`) is therefore **update-first** so the singleton card is edited in place and never flickers: it reads the persisted resident id (`_get_resident_task_card`, `telegram/manager.py:1812`) and, when one exists, edits that resident through Telegram (`_try_update_progress_message`, `telegram/manager.py:1404`) and returns the **same** compound id — sending nothing new and deleting nothing. `_deliver_channel_frame` (`telegram/manager.py:1511`) acquires one stable per-account+chat transaction lock before compose/read/supersede/send/persist/delete/channel-commit, serializing automatic heartbeats/finalize with programmable watches; it commits `_task_card_channels` only after an applied edit, identical-content no-op, first send, successful rotation, or successful replacement. Current kernel update/finalize reverse calls repeat the original turn route (`account` + `chat_id`) alongside the card id; the manager requires an exact match, while legacy/private callers that omit those fields retain compound-id routing for compatibility. Before editing a resident it checks `_resident_superseded` (`telegram/manager.py:1747`), which returns true **only** when the account's high-water mark (`_get_last_message_id`, `telegram/manager.py:1728`) deterministically proves a newer chat message id sits below the resident; the caller first rejects malformed or cross-bound resident ids before any transport, while an unknown latest id (`None`) returns false so a correctly bound resident falls through to the in-place edit and never deletes on uncertain state (preserves #891). When superseded, `_rotate_task_card_to_latest` (`telegram/manager.py:1769`) rotates the card back to the bottom **create-new-first**: it sends a fresh card, persists the new id as resident, and only then deletes the exact old resident (`_delete_task_card_message`) — never a normal message. This ordering never leaves the chat with zero cards: a failed replacement send keeps the old card/id and deletes nothing; after a successful send, old-card deletion is allowed only when `_set_resident_task_card` acknowledges the new resident write. A persistence failure retains both visible cards and surfaces `resident_persist_failed`; an old-card delete failure keeps the new latest card as resident and surfaces `stale_delete_failed`. Both partial outcomes retain the new id while automatic and programmable consumers make the fault observable. Telegram's exact `message is not modified` response is a successful no-op that retains the id; unknown/transient edit failures return an error and preserve resident plus committed slot state. Only Telegram's allowlisted, provider-confirmed `message to edit not found` / `message can't be edited` conditions call `_recover_task_card_by_replacement` (`telegram/manager.py:1699`), which likewise sends the replacement first, then persists the new id and best-effort deletes the exact stale `account:chat:message`. `_task_card_finalize` (`telegram/manager.py:1874`) freezes the card on its concrete last batch (rows + `✓` markers + final elapsed) with no generic overall `DONE` subject and, because the durable final reply has by then pushed the card off the bottom, rotates it to the last position by the same rule; the legacy scalar form keeps `✅ TASK CARD · DONE`.
- **Task Card render: rows, heartbeat elapsed, fixed footer, one card-level time line.** `_format_task_card_text` (`telegram/manager.py:1951`) renders the batched multi-row form via `_format_rows_task_card_text` (`telegram/manager.py:1994`): one line per tool call (`tool.action`, redacted reasoning, own whole-second elapsed via `_format_elapsed` at `telegram/manager.py:2126`, `✓` when done) with **no** per-row inline timestamp. The card carries a single card-level time line instead (Jason #6894/#6899, #7213/#7216): the bare stamp `HH:MM:SS UTC±HH` with **no label** (`_TASK_CARD_TIME_PREFIX` is the empty string at `telegram/manager.py:91`) rendered as the card's **final standalone line after the footer**, sourced from the first non-empty `started_at` in original row order and omitted entirely when no row carries a usable stamp; its exact text is counted in the reasoning-excerpt budget. Redaction runs on each row before any excerpt/trim, and every row always stays represented (rows are never dropped; only per-row reasoning excerpts shrink). The `_TASK_CARD_TEXT_LIMIT` (3500) budget bounds that reasoning-excerpt shrinkage only, **not** the whole render: fixed per-row scaffolding is unbounded in row count, so a very large operator-set `LINGTAI_TASK_CARD_MAX_TOOL_ROWS` can push the render above the budget (and above Telegram's transport limit) — by design the code neither drops requested rows nor truncates the final string. Both the running and frozen renders carry the fixed `_TASK_CARD_FOOTER` (`telegram/manager.py:83`, "Don't reply to this Task Card. Use /taskcard on|off to toggle."). The kernel owns the batch/timer: BaseAgent's pre-dispatch hook builds one row per call and starts a 0.5s monotonic heartbeat (elapsed floored to whole seconds, so half-second frames read 0s, 0s, 1s, 1s, 2s), captures each tool's local start instant **once** into an immutable `started_at` string (`_capture_task_card_started_at`/`_format_task_card_timestamp` at `base_agent/__init__.py:2458,2562`, separate from the monotonic elapsed clock, so heartbeats never change it and parallel rows keep their own), the result hook freezes the completed row, and the payload the reverse channel carries is the `rows` list (see `base_agent/__init__.py` and `src/lingtai/tools/mcp/ANATOMY.md`).
- **Resident card, two composed slots — `automatic` + `programmable` (Jason #7258/#7259).** The ONE resident message per account+chat is composed from two independent slots kept in `_task_card_channels` (`telegram/manager.py:432`, keyed `"{account}:{chat_id}"`). `_handle_task_card_update` reads `channel` (default `automatic`); an unknown channel is a hard error, and `channel == "programmable"` routes to `_task_card_programmable` (`telegram/manager.py:1914`) before the automatic create/update/finalize dispatch. Each write stores only its own slot via `_set_channel_frame` (`telegram/manager.py:1456`; `None` clears just that slot) and re-renders through `_compose_channels` (`telegram/manager.py:1467`) — programmable-empty composes byte-for-byte to the automatic frame (no regression); with both present the programmable block is appended under a `— WATCH —` header. So updating either slot preserves the other, and programmable `finalize` clears only the programmable slot while the automatic slot and the message itself remain. The programmable slot is the public kernel `task_card` controller's output: the controller sends only a validated card object (never code), and `_format_programmable_card_text` (`telegram/manager.py:1600`) renders it English-only with the same secret redaction and reasoning cap as the automatic path. The controller and its watch lifecycle live in `src/lingtai/kernel/task_card_controller.py` (see `base_agent/ANATOMY.md`). After channel/sub-action validation, `_handle_task_card_update` checks the service-owned agent-wide flag; when false it returns an explicit non-error suppression result before transport, resident-id, or composed-slot mutation, so both automatic mechanics and programmable watches keep running. Telegram message previews, structured notification items, and check/read/search items derive the same current boolean at projection time rather than rewriting stored message records (`telegram/manager.py`).
