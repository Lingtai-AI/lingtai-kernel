# lingtai.mcp_servers

Curated and built-in MCP server package implementations shipped inside the `lingtai` Python distribution. They are launched by catalog/script entry points such as `python -m lingtai.mcp_servers.<name>` and expose real addon tools (IMAP, Telegram, Feishu, WeChat, WhatsApp, Cloud Mail), daemon lifecycle tools, plus bundled progressive-disclosure manuals.

## Components

| File / folder | Role |
|---|---|
| `_skill.py` | Shared bundled-skill helper: re-exports the kernel-owned `split_frontmatter` from `lingtai.kernel._frontmatter` (one impl shared with the prompt-section catalog; kernel never imports the wrapper), `load_skill()` loads package `SKILL.md`, `manual_action_description()` injects frontmatter into the schema, and `manual_payload()` returns the manual body + absolute path without sidecar lists (`_skill.py:36-79`). |
| `_identity.py` | Shared public-identity envelope/path/write helper for curated messaging MCPs: builds the `lingtai.mcp.identity.v1` document, computes `system/mcp_identities/<name>.json`, and performs the newline-terminated atomic JSON write. Provider-specific account fields and redaction stay in each provider. |
| `daemon_common/` | Built-in daemon lifecycle MCP. `daemon_common/server.py:1-151` exposes `finish(status, summary?, reason?, artifacts?)`, validates the call, and atomically writes the internal per-run `daemon_completion.json` file named by `LINGTAI_DAEMON_COMPLETION_FILE`; daemon runners validate that file before allowing success. |
| `telegram/`, `imap/`, `feishu/`, `wechat/`, `whatsapp/`, `cloud_mail/` | Curated messaging MCPs. TelegramManager requires an injected `NotificationStorePort`; `telegram/server.py` constructs one POSIX adapter, and handled-mirror policy runs against the current payload in one compare-update so newer mirrors survive (`src/lingtai/mcp_servers/telegram/manager.py:415-425`, `src/lingtai/mcp_servers/telegram/manager.py:1239-1281`, `src/lingtai/mcp_servers/telegram/server.py:655-663`). The external LICC path/envelope and persistent-message lanes remain unchanged. |
| `telegram/task_card/` | Telegram-owned **programmable Task Card** unit (governed component with paired `ANATOMY.md`/`CONTRACT.md` + co-located `SKILL.md`). The model-facing `task_card` tool drives the resident card's *programmable* slot: it runs an agent-supplied Python renderer and projects only validated data onto the private `_lingtai_telegram_task_card` reverse channel with `channel="programmable"`. `TelegramManager` stays the single render/compose/persistence owner. The wrapper `Agent` wires it as the Composition Root once a Telegram client exists; the controller depends on a narrow host `Protocol` (`interface.py`), not the concrete `Agent`. See `src/lingtai/mcp_servers/telegram/task_card/ANATOMY.md`. |
| Per-package `SKILL.md` | The human/agent-facing bundled manual. If a manual has sidecars, the sidecar inventory and relative paths live in this markdown, not in the tool payload. |
| `pyproject.toml` package-data entries | Ships every curated MCP `SKILL.md`; `reference/**/*` and `assets/**/*` are also packaged for future sidecar files (`pyproject.toml:81-86`). |

## Connections

- Catalog/script launchers (`pyproject.toml:43-49`) start these servers as subprocess MCPs; agents activate them through the generic MCP capability (`src/lingtai/tools/mcp/ANATOMY.md`).
- Manager schemas include `manual` in each action enum and use `_skill.manual_action_description()` to advertise the bundled skill without loading the full body into the resident schema.
- Tests pin the manual contract, package-data sidecar support, and Telegram parity in `tests/test_mcp_skill_manuals.py` and `tests/test_telegram_rich_formatting.py`.

## Composition

Parent: `src/lingtai/` wrapper package (`src/lingtai/ANATOMY.md`). Sibling wrapper areas include `agent.py`, `core/`, `services/`, and `intrinsic_skills/`. Curated MCPs are independent subprocess packages, not intrinsic capabilities.

## State

The package itself is mostly code + packaged manuals. Runtime state is per-agent and server-specific: e.g. message caches, contacts, inbox replay guards, credential-derived identities, or daemon completion sentinels live under the agent workdir/run dir or `.secrets/`, not in `src/lingtai/mcp_servers/`. The shared manual and identity helpers have no persistent state of their own. Telegram's per-account non-secret `state.json` (under `<workdir>/telegram/<alias>/`) additionally persists the resident Task Card id per chat under a `task_cards` map (`{chat_id: compound_id}`), loaded backward-compatibly and normalized against malformed values by `TelegramAccount._normalize_task_cards`, so the one tracked resident target per account+chat survives a refresh; it is not proof that historical unknown orphan cards do not exist. The same account additionally owns an **ephemeral, non-persisted** per-chat high-water mark of the latest observed Telegram message id (`_last_message_ids`, `telegram/account.py:190`), bumped by every inbound plain message the poll loop sees (`_process_update`) and every outbound `send_message`/`send_photo`/`send_document`, but never by an edit or delete; `_note_chat_message_id`/`get_last_message_id` (`telegram/account.py:1208,1235`) record and expose it, guarded by the same `_state_lock` as `task_cards`. Because Telegram message ids are monotonic per chat this mark is "the chat's last message id"; the manager reads it to keep the resident Task Card residing as the last message (Jason #5272/#5273/#5275). It is deliberately not persisted, so a refresh reports `None` (unknown) until the next observed message, and the manager treats an unknown latest-message mark conservatively by editing in place; malformed or cross-bound resident ids fail loud before transport, and neither condition authorizes deletion. Separately, `TelegramService` owns one current-agent delivery boolean in `<workdir>/telegram/taskcard.json`, loaded with a compatibility default of `True` and written through the kernel atomic JSON primitive with fsync; bound getter/setter callbacks let every configured account's local `/taskcard` command share that one value (`telegram/service.py`, `telegram/account.py`).

## Notes

- **Notification contract:** curated messaging MCPs that change structured notification metadata (`recent_messages`, `latest_incoming`, `referenced_messages`, stable IDs, routing hooks, or preview/body placement) must check `src/lingtai/services/LICC_NOTIFICATION_CONTRACT.md` in the same change.
- **Manual sidecar minimal contract:** `action="manual"` returns the main `SKILL.md` body, parsed metadata, and the main `SKILL.md` absolute `path` only. Concrete `assets/` and `reference/` lists MUST NOT be returned as structured tool fields; `SKILL.md` is the single source of truth for what sidecars exist and how to follow their relative paths.
- **Packaging discipline:** when adding manual sidecars, put their relative paths in `SKILL.md` and keep the package-data globs for `reference/**/*` / `assets/**/*` so wheels contain them (`pyproject.toml:81-86`).
- **Telegram private reverse-channel tool:** `telegram/server.py` `list_tools` advertises only the public `telegram` tool (validated against `SCHEMA` by the mcp library's default `validate_input=True`). Its `build_server._call_tool` also accepts one **unlisted** private tool name, `_PRIVATE_TASK_CARD_TOOL = "_lingtai_telegram_task_card"` (`telegram/server.py:66`), used only by internal LingTai code — the kernel's automatic-card driver and the Telegram-owned programmable `task_card` controller — to project the live Task Card, never by the model; being unlisted, it skips public-schema validation yet still reaches the handler, which forces `action="_task_card_update"` before `manager.handle` so the hidden route cannot invoke any public action. The mechanism, internal callers (`_TASK_CARD_TOOL`), and regression tests are described in `src/lingtai/tools/mcp/ANATOMY.md`.
- **Tracked resident Task Card target, composed per account+chat and kept at the last message (Jason #6665/#6667, #6894/#6899, #5272/#5273/#5275).** `_handle_task_card_update` (`telegram/manager.py:1717`) dispatches create/update/finalize through `_deliver_channel_frame` (`telegram/manager.py:1576`), which takes one stable per-account+chat transaction lock across automatic and programmable callers, re-reads the tracked resident inside that transaction, and commits composed channel state only after transport success. The durable `task_cards` map names one exact bot-owned resident target; it is not a scan or proof of global chat-history cardinality, and unknown historical orphan cards are never guessed at or deleted. If the resident is already the account's latest observed message, the manager edits it in place; an unknown high-water (`_resident_superseded`, `telegram/manager.py:1874`) remains conservative and never authorizes deletion. If a newer message deterministically supersedes it, `_rotate_task_card_to_latest` (`telegram/manager.py:1896`) uses the last committed render for a same-content edit/no-op existence probe on that exact resident id when available; after a cold in-memory start, the exact delete outcome itself is the existence/removal probe (`_replace_task_card_after_probe`/`_delete_task_card_message_outcome`, `telegram/manager.py:1820,1943`). Unknown/transient edit-probe errors fail closed. The exact old id must then be confirmed deleted, or Telegram must explicitly report it already missing, before any replacement send; unknown or permission delete failure surfaces `stale_delete_failed` and blocks the send (never a successful partial). This old-first ordering prevents tracked rotation from intentionally displaying two cards, while a replacement-send failure after confirmed deletion may leave zero and reports `old_resident_deleted`. A new resident write failure retains the sole new in-process id and surfaces successful-partial `resident_persist_failed`. Provider-confirmed edit-impossible recovery (`_recover_task_card_by_replacement`, `telegram/manager.py:1804`) follows the same delete/missing-confirm-before-send rule. Identical edit content is a successful no-op. Finalize uses the same transaction after the durable final reply pushes the card off the bottom; legacy scalar finalize keeps its historical done form. The per-chat last-message high-water mark is an ephemeral, int-only observation owned by `TelegramAccount` (`_note_chat_message_id`/`get_last_message_id`, `telegram/account.py`; rejects `bool`/non-`int`, not persisted, `None`-conservative after refresh).
- **Task Card render: rows, heartbeat elapsed, dynamic footer, bounded metadata, render-time current-time line.** `_format_task_card_text` (`telegram/manager.py:2126`) renders the batched multi-row form via `_format_rows_task_card_text` (`telegram/manager.py:2250`): one line per tool call (`tool.action`, redacted reasoning, own whole-second elapsed via `_format_elapsed` at `telegram/manager.py:2423`, `✓` when done) with **its own** inline `started_at` suffix — each row shows the exact captured instant for *that* row, never the first row's or any other row's. Malformed/missing `started_at` degrades to no inline suffix rather than raising. The card's final standalone line is always present (unlike the retired started_at-derived line, it never depends on any row carrying a stamp): `Current Time: HH:MM:SS UTC±HH` (`_TASK_CARD_TIME_PREFIX`, `telegram/manager.py:103`), sourced from the **render instant** via `_task_card_render_time`/`_format_task_card_current_time` (`telegram/manager.py:2355`, `:115`) — never any row's start instant — and injectable via the `now` keyword for deterministic tests; its exact text is counted in the reasoning-excerpt budget. Redaction runs on each row before any excerpt/trim, and every row always stays represented (rows are never dropped; only per-row reasoning excerpts shrink). The `_TASK_CARD_TEXT_LIMIT` (3500) budget bounds that reasoning-excerpt shrinkage only, **not** the whole render: fixed per-row scaffolding is unbounded in row count, so a very large operator-set `LINGTAI_TASK_CARD_MAX_TOOL_ROWS` can push the render above the budget (and above Telegram's transport limit) — by design the code neither drops requested rows nor truncates the final string. Both running and frozen renders carry `_TASK_CARD_FOOTER` (`telegram/manager.py:91`) plus a live `current: X` suffix built by `_task_card_footer` (`telegram/manager.py:106`) from the `normal_rows` keyword — the manager's three call sites (`_task_card_create`/`_task_card_update`/`_task_card_finalize`) pass `self._taskcard_normal_rows()` (`telegram/manager.py:493`), which reads `TelegramService.taskcard_normal_rows()` (validated `1-10`, defaulting to `1` for narrow test/third-party service doubles that omit the getter) — so the footer always names the actual current operator setting, e.g. `/taskcard N sets normal rows (1-10, current: 3).`; immediately after it, `_format_task_card_metadata` (`telegram/manager.py:2197`) renders at most two lines / 150 characters from numeric session cache rate, cumulative cache miss/budget, API calls, and context usage, before the final Current-Time line. The kernel owns the batch/timer: BaseAgent's pre-dispatch hook builds one row per call and starts a 0.5s monotonic heartbeat (elapsed floored to whole seconds, so half-second frames read 0s, 0s, 1s, 1s, 2s), captures each tool's local start instant **once** into an immutable `started_at` string (`_capture_task_card_started_at`/`_format_task_card_timestamp` at `src/lingtai/kernel/base_agent/__init__.py:2475,2612`, separate from the monotonic elapsed clock, so heartbeats never change it and parallel rows keep their own), the result hook freezes the completed row, and the payload the reverse channel carries is the `rows` list plus a numeric-only canonical session metadata projection (see `base_agent/__init__.py` and `src/lingtai/tools/mcp/ANATOMY.md`).
- **Resident card, two composed slots — `automatic` + `programmable` (Jason #7258/#7259).** The one tracked resident target per account+chat is composed from two independent slots kept in `_task_card_channels` (`telegram/manager.py:443`, keyed `"{account}:{chat_id}"`). `_handle_task_card_update` reads `channel` (default `automatic`); an unknown channel is a hard error, and `channel == "programmable"` routes to `_task_card_programmable` (`telegram/manager.py:2036`) before the automatic create/update/finalize dispatch. Each write stores only its own slot via `_set_channel_frame` (`telegram/manager.py:1526`; `None` clears just that slot) and re-renders through `_compose_channels` (`telegram/manager.py:1537`) — programmable-empty composes byte-for-byte to the automatic frame (no regression); with both present the programmable block is appended under a `— WATCH —` header. So updating either slot preserves the other, and programmable `finalize` clears only the programmable slot while the automatic slot and the message itself remain. When the programmable slot is the ONLY resident content, clearing it would compose empty text (which Telegram cannot edit to); finalize instead delivers a stable nonempty `_TASK_CARD_WATCH_STOPPED` (`— WATCH STOPPED —`, `telegram/manager.py:1521`) terminal marker while still committing the slot clear only after delivery, so the tracked resident target stays reusable and the marker never becomes stored channel state (`_deliver_channel_frame`'s `empty_fallback`). The programmable slot is the public Telegram-owned `task_card` controller's output: the controller sends only a validated card object (never code), and `_format_programmable_card_text` (`telegram/manager.py:1689`) renders it English-only with the same secret redaction and reasoning cap as the automatic path. The controller and its watch lifecycle live in the Telegram-owned unit `src/lingtai/mcp_servers/telegram/task_card/` (paired `ANATOMY.md`/`CONTRACT.md`, co-located `SKILL.md`; see `src/lingtai/mcp_servers/telegram/task_card/ANATOMY.md` and `base_agent/ANATOMY.md`). After channel/sub-action validation, `_handle_task_card_update` checks the service-owned agent-wide flag; when false it returns an explicit non-error suppression result before transport, resident-id, or composed-slot mutation, so both automatic mechanics and programmable watches keep running. The one targeted exception: a hidden programmable `finalize` still clears its committed programmable slot internally (no transport), so stopping a suppressed watch cannot resurface a stale frame after `/taskcard on`; ordinary hidden create/update stay non-committing. Telegram message previews, structured notification items, and check/read/search items derive the same current boolean at projection time rather than rewriting stored message records (`telegram/manager.py`).
