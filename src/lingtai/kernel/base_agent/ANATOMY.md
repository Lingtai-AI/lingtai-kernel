# base_agent

> **Maintenance:** see the `lingtai-kernel-anatomy` skill. **Coding agents** update this file in the same commit as code changes. **LingTai agents** report drift as issues/mail/PR proposals; do not silently fix.

Generic agent kernel. Single class `BaseAgent` with methods distributed across 6 helper modules. `__init__.py` retains the constructor, properties, state machine, and subclass-overridable hooks.

> **History note:** there used to be a 7th module, `soul_flow.py`. It was deleted in `1acd183` — soul-domain logic moved to `intrinsics/soul/flow.py`, and the wire-splice logic that lived in it became `tc_inbox.TCInbox.drain_into()`. Mention preserved here only because old patches/discussions reference it.

## Components

- `base_agent/__init__.py` — `BaseAgent` coordinator. Its constructor requires injected `WorkdirLeasePort` and `NotificationStorePort`, stores the latter as `_notification_store`, and performs no concrete notification-adapter construction (`src/lingtai/kernel/base_agent/__init__.py:304-368`). `_sync_notifications` reads snapshots/fingerprints through that Port and preserves the existing state/wake/live-holder order (`src/lingtai/kernel/base_agent/__init__.py:1195-1470`).

- `base_agent/identity.py` — Identity, manifest, and status (~301 lines). `_set_name` (`base_agent/identity.py:13`), `_set_nickname` (`base_agent/identity.py:26`), `_update_identity` (`base_agent/identity.py:32`), `_LLM_PUBLIC_KEYS` safelist constant (`base_agent/identity.py:57`), `_build_manifest` (`base_agent/identity.py:60`), `_safe_llm_from_service` (`base_agent/identity.py:102`) — surfaces a sanitized `llm` block (provider/model/base_url, safelisted) from the live `LLMService` instance for `.agent.json` and the identity prompt section (issue #78). Never reads from `init.json` (kernel stays preset-oblivious); preset metadata is added by the wrapper's `Agent._build_manifest` override. `_status` (`base_agent/identity.py:182`).
- `base_agent/lifecycle.py` — Heartbeat, signal-file detection, start/stop, refresh, hidden IDLE→ASLEEP timeout, nudge dispatch (1170 lines). **Token restore (agent-session rebuild)**: on `_start`, after chat-history restore, `agent.rebuild_agent_session()` rebuilds the current molt generation's since-molt token aggregate from the durable trajectory (optimized indexed/reverse-scan path) and `restore_token_state` is **seeded from that rebuilt aggregate** — so a refresh/restart preserves since-current-molt `token_usage.session` totals instead of restoring LIFETIME `token_ledger.jsonl` totals (the #679-class defect). The lifetime `sum_token_ledger` restore remains only as a back-compat fallback when the event-based rebuild is unavailable (brand-new agent, tier `none`, or an unexpected error). `_start` (`base_agent/lifecycle.py:100`), `_stop` (`base_agent/lifecycle.py:196`), `_shutdown_daemon_runtime` (`base_agent/lifecycle.py:242`), `_start_heartbeat` (`base_agent/lifecycle.py:273`), `_stop_heartbeat` (`base_agent/lifecycle.py:287`), `_heartbeat_loop` (`base_agent/lifecycle.py:313`), `_perform_refresh` (`base_agent/lifecycle.py:612`), `_can_fallback_preset` (`base_agent/lifecycle.py:1121`), `_check_rules_file` (`base_agent/lifecycle.py:1135`). **Daemon teardown contract**: `_stop` calls `_shutdown_daemon_runtime` after joining the main loop but before closing session services, stopping heartbeat, or releasing the workdir lock. If the daemon capability is present, `DaemonManager.shutdown_for_agent_stop()` kills tracked CLI process groups and shuts down executor pools while `.agent.heartbeat` is still fresh, preventing refresh watchers from seeing a dead heartbeat/clear lock while the old Python interpreter is still held alive by daemon workers. **Heartbeat-file lifetime contract**: `.agent.heartbeat` is created on `_start_heartbeat` and unlinked at the **end** of `_stop` (just before the workdir-lease `release`), so external observers — TUI launcher, `lingtai-tui list`, `lingtai-tui purge` — see the workdir as live for the entire teardown window. The heartbeat loop intentionally keeps writing fresh timestamps after `agent._shutdown.is_set()` (loop predicate is `agent._heartbeat_thread is not None`); only signal-file detection is gated on `_shutdown`. Without this contract, a quick suspend→relaunch would race a still-living interpreter into the same workdir (the kernel's flock prevents data corruption but two Python processes still appear in `ps`). **Nudge dispatch**: just before the notification sync poll, the heartbeat loop calls `nudge.run_checks(agent)` (`base_agent/lifecycle.py:~490`), wrapped in `try/except` that logs warnings. Each registered nudge check (today: `kernel_version`, `source_drift`, and `goal`) throttles itself internally and writes to its notification surface through `nudge.upsert` / `nudge.remove` or the goal/system channel. See `nudge/ANATOMY.md`. **Hidden idle timeout + notification sync poll**: before notification sync, `_maybe_sleep_after_idle_timeout` moves agents that stayed IDLE for `IDLE_SLEEP_TIMEOUT_SECONDS` (24h) to ASLEEP, saving chat history and logging `idle_sleep_timeout`; this replaces the old agent-visible stamina countdown and is not surfaced through init.json, prompt, status, or tool metadata. The heartbeat loop then calls `agent._sync_notifications()`, wrapped in a `try/except` that logs warnings without aborting the tick. This is the primary trigger for `.notification/` filesystem reads (including the `nudge.json` file the dispatch may have just written). **Refresh dispatch contract**: `_perform_refresh` calls `agent._build_launch_cmd()` as a bound method — must NOT be re-introduced as a module-level function (would shadow the `Agent` subclass override and silently no-op every user refresh). See issue #7 — confirmed in vivo against deepseek_pro 2026-05-05 (`refresh_start → refresh_no_launch_cmd` in 0.78ms, PID stable across 5 "refreshes"). **Refresh handshake self-sufficiency**: `_perform_refresh` is called from three sites — the heartbeat (`.refresh` signal-file path), the `system(action='refresh')` intrinsic, and the AED preset-fallback in `turn.py`. All three reach the same code without coordination, so `_perform_refresh` normalizes the on-disk handshake (`.refresh.taken` ack file present and `.refresh` cleared) before spawning the watcher subprocess, and sets `agent._cancel_event` + `agent._shutdown` after spawn so the lock-release phase completes regardless of caller. The `refresh_deferred_relaunch` event records `handshake` ∈ {`preexisting_taken`, `renamed_refresh`, `synthesized_direct_call`, `synthesized_after_rename_failed`, `ack_write_failed`} for telemetry; if the ack cannot be established, `_perform_refresh` logs `refresh_ack_failed` and returns before spawning a watcher or setting shutdown. Direct callers thus never trip the watcher's `phase='ack'` or `phase='lock'` timeouts in the normal established-ack path. **Refresh stale-process recovery**: if the watcher relaunch is rejected by the CLI duplicate-process guard (`another lingtai agent is already running`) and the blocking process is still a same-workdir `lingtai run` with no fresh heartbeat, the watcher logs `refresh_watcher_stale_duplicate_*`, sends `SIGTERM`, escalates to `SIGKILL` after a short grace period, and retries. A fresh heartbeat blocks cleanup and logs `refresh_watcher_duplicate_alive` instead. **Terminal refresh failure visibility**: when all `MAX_ATTEMPTS` relaunch attempts are permanently exhausted, the watcher makes the dead agent diagnosable rather than silently gone — `_publish_refresh_failed_permanent` writes a bounded `logs/refresh_failed_permanent.json` artifact and appends a high-priority `refresh_failed_permanent` event (`source="refresh"`, `ref_id="refresh_failed_permanent"`) to `.notification/system.json`, both carrying the same `failure_state` metadata (attempts, last/duplicate/relaunch PID, heartbeat age+status, bounded stderr tail, cleanup action/result/error, relaunch error, stderr-log path, and `recovery_guidance`). A final `refresh_failed_permanent` event is also logged to `logs/events.jsonl`. It does not delete `.agent.lock` or change retry/restart semantics. **Refresh watcher event redaction**: the watcher subprocess persists its own `refresh_watcher_*` events to `logs/events.jsonl` through an inline `log()` (`base_agent/lifecycle.py:~742`) that runs in a separate process and therefore does **not** pass through `CompositeLoggingService.log` / `redact_for_trajectory`. That inline `log()` imports `lingtai.kernel.trace_redaction.redact_for_trajectory` (the whole-entry redactor, not just `redact_text` value-walking) and applies it to the full event dict before the JSONL write, so secret-shaped values reaching `stderr_tail` (relaunched-process stderr, e.g. a config traceback echoing a token), `cmdline`, and `error` previews are redacted to `<REDACTED:*>` placeholders before persistence — and, for key-aware parity with normal trajectory logging, values under secret-named keys are removed even when they do not match a known token shape. The redaction is fail-open for relaunch reliability but **not silent**: if the redactor cannot be imported the event carries a non-secret marker `redaction_unavailable=True`, and if `redact_for_trajectory` raises at apply time the event is rebuilt from only known-safe scalar fields (`type`/`address`/`agent_name`/`ts`) with `redaction_unavailable=True` and `redaction_error=True` — the unredacted `**kw` fields are dropped rather than persisted raw. The watcher path thus matches the in-process redaction contract documented in the root anatomy's `trace_redaction.py` entry while making degradation diagnosable; the shared terminal-failure metadata builder likewise redacts the bounded `last_stderr_tail` before writing `refresh_failed_permanent.json`, `.notification/system.json`, or the final `refresh_failed_permanent` event; if the watcher subprocess cannot import the redactor or the redactor raises, it marks `redaction_unavailable` and replaces the stderr tail with a `REDACTED` sentinel instead of leaking raw stderr.
- `base_agent/turn.py` — Turn engine: main loop, message dispatch, AED, response processing, and ACTIVE-turn tool-call progress-meter wiring (1670 lines). Transient provider retry helpers (`base_agent/turn.py:33-126`) classify SDK/network 5xx-ish failures and `EmptyLLMResponseError` before they spend AED budget. `WorkerStillRunningError` (from `llm_utils.py`) is handled directly in the AED loop: the worker interface is marked poisoned (see `worker_recovery.py`), a bounded/redacted recovery artifact is written, a high-priority `kernel.llm_worker_hang` notification is published, a skip-save forced refresh is requested, and the agent is put STUCK→ASLEEP with `skip_post_turn_save=True` so the in-process `ChatInterface` is never mutated while the worker future is still alive — refresh/relaunch brings up a fresh interface from the last safe on-disk history. A residual poison guard at the top of the AED `try` also short-circuits any later turn that reaches the loop while poisoned. No filesystem sentinel — the previous `.llm_hang` watchdog/sentinel system was removed because its 120s timer false-fired on slow thinking-model turns and the sentinel could block wake-from-asleep even after a successful refresh. Run loop: `_run_loop` (`base_agent/turn.py:509`), `_concat_queued_messages` (`base_agent/turn.py:830`), `_handle_message` (`base_agent/turn.py:876`), `_handle_request` (`base_agent/turn.py:959`), `_handle_tc_wake` (`base_agent/turn.py:1014`) — **post-redesign: drives one inference round off the existing canonical interface via `session.send(None)` when the tail is `user[ToolResultBlock]` (the shape produced by `_inject_notification_pair`).** Legacy `tc_inbox.drain()` items are still spliced in the old way (call appended here, result passed through send) at the top of the function for back-compat, but the queue is always empty in production. The wire-drive path is what makes notifications visible to the LLM: no fake user message, no meta prefix. **AED state transitions** (within `_run_loop`): transient provider failures first get up to three local retry attempts with 1/2/4s backoff, pending-tool-call healing, and `system.stuck_revive` retry messages, but without incrementing `aed_attempts` or rebuilding the session (`base_agent/turn.py:683-706`). Counted AED attempts then close pending tool calls with synthesized results (`base_agent/turn.py:730-734`), set STUCK (`base_agent/turn.py:735`) before rebuilding the session (`base_agent/turn.py:775`), and set ACTIVE again right before re-entering `_handle_message` with the recovery message (`base_agent/turn.py:779`). Without that ACTIVE flip, a recovered turn that does long work (file writes, mail, bash) leaves `.agent.json:state` reading "stuck" until the post-turn sweep at `base_agent/turn.py:781-810` finally writes `sleep_state` (IDLE) — visible to the TUI/portal as a stale "滞" badge while the agent is in fact busy. `_get_guard_limits` (`base_agent/turn.py:1215`), `_process_response` (`base_agent/turn.py:1378`; empty response raises at `base_agent/turn.py:1426`). **Tool errors stay in the normal result flow**: `_process_response` no longer has a repeated-identical-tool-error hard-stop or `session.send(str)` continuation path. Tool execution errors remain ordinary `ToolResultBlock` payloads that are sent back through `session.send(tool_results)`; `ToolExecutor` enriches those error payloads with model-visible `tool_error` recovery metadata (`tool_executor.py:146-235`) while the generic tool-loop / emergency limits remain the only backstop for true infinite loops.
- `base_agent/worker_recovery.py` — WorkerStillRunning poison/recovery helpers (design ref: PR #298). When the LLM worker is still alive after timeout + grace, the in-process `ChatInterface` is poisoned for this process and recovery happens from durable on-disk state after refresh/relaunch. `build_worker_hang_context` captures safe message/exception metadata (never the poisoned interface); `write_worker_hang_artifact` writes a bounded/redacted `history/unfinished_turns/worker_still_running_*.json` (no raw chat, tool args, or tool results); `mark_worker_interface_poisoned` sets process-local poison state (`agent._llm_worker_interface_poisoned` et al.); `is_worker_interface_poisoned` is the guard predicate read by `turn.py`/`__init__.py`; `publish_worker_hang_notification` publishes a high-priority `kernel.llm_worker_hang` / `worker_still_running:*` system event (idempotent on ref_id); `request_worker_hang_refresh` calls `_perform_refresh(skip_chat_history_save=True, ...)` once (idempotent); `rehydrate_worker_hang_recovery` (called from `lifecycle._start`) re-surfaces open artifacts as notifications after restart; `maybe_prepend_worker_hang_recovery_prompt` (called from `_handle_request`) prepends one concise recovery notice to the next safe text request.
- `base_agent/tools.py` — Tool surface: schemas, dispatch, registry. `_dispatch_tool` (`base_agent/tools.py:14-33`), `_refresh_tool_inventory_section` (`base_agent/tools.py:36-50`) appends the selected-language glossary body to each tool's description via `tool_glossary.append_tool_glossary`, `_build_tool_schemas` (`base_agent/tools.py:53-101`) injects the standard `reasoning` property into every tool schema, `_add_tool` (`base_agent/tools.py:104-132`), `_remove_tool` (`base_agent/tools.py:135-143`), `_override_intrinsic` (`base_agent/tools.py:143-155`), `_has_capability` (`base_agent/tools.py:158-160`).
- `base_agent/prompt.py` — System prompt building, flushing, updating (80 lines). `_build_system_prompt` (`base_agent/prompt.py:26-37`), `_build_system_prompt_batches` (`base_agent/prompt.py:40-56`), `_flush_system_prompt` (`base_agent/prompt.py:59-66`), `_update_system_prompt` (`base_agent/prompt.py:69-80`).
- `base_agent/messaging.py` — mail arrival and system-event production. `_enqueue_system_notification` makes its dedupe/append decision from the current `system` payload in one Store compare-update and consumes event identity from `CompareUpdateResult.value`; no caller-held channel lock remains (`src/lingtai/kernel/base_agent/messaging.py:66-207`).

## Connections

- All submodules are leaves — they import nothing from `base_agent/`. Cross-module communication is mediated through the `agent` parameter (the BaseAgent instance).

- `__init__.py` imports from all submodules lazily (inside method bodies) to avoid circular imports at load time. Soul-flow pass-throughs resolve the soul intrinsic through the injected registry (`BaseAgent._intrinsic_hook("soul", ...)`), no longer by import — the kernel cannot import `tools`.
- `__init__.py` → the required `NotificationStorePort` for sync snapshot/fingerprint and → `notifications.py` for Core policy.
- `__init__.py` constructs `SessionManager` without notification-specific send hooks; notification delivery is owned by `_sync_notifications` plus the ACTIVE canonical-payload attach step in `turn.py`.
- `__init__.py` → `event_journal.EventJournalPort`: `_log` assembles event fields and invokes `append()` (`base_agent/__init__.py:938-946`); lifecycle shutdown invokes `close()` best-effort after session/mail teardown and before the final manifest write, heartbeat stop, and workdir-lease release (`base_agent/lifecycle.py:266-292`).
- `__init__.py` / `lifecycle.py` → `workdir_lease.WorkdirLeasePort`: the constructor stores the required Port and acquires it once with a 10s grace (`base_agent/__init__.py:349-358`); the signature-preserving rollback decorator releases best-effort only after successful acquisition and re-raises the original construction failure (`base_agent/__init__.py:254-268`). `_stop` releases it last, after the final manifest write and heartbeat stop (`base_agent/lifecycle.py:284-292`). See `src/lingtai/kernel/workdir_lease/ANATOMY.md`.
- `lifecycle.py` → soul hooks (resolved via `BaseAgent._intrinsic_hook`) (via `agent._start_soul_timer()`, `agent._cancel_soul_timer()`), soul inquiry hook (via `agent._run_inquiry()`), and `agent._sync_notifications()` on every heartbeat tick.
- `turn.py` → soul hooks (via `agent._cancel_soul_timer()`), soul inquiry hook (via `agent._run_inquiry()`). The `tc_inbox` drain calls in `_handle_request`, the pre-request hook, and `_handle_tc_wake` are still wired but always find an empty queue under the `.notification/` redesign.
- `messaging.py` → `notification_store.NotificationStorePort.compare_update_channel`; producer envelope and dedupe/priority policy stay in Core.
- `turn.py`, `lifecycle.py`, and `__init__.py` → `worker_recovery.py` for WorkerStillRunning poison state, unfinished-turn artifacts, refresh request, startup rehydration, and the one-shot request-prompt notice. Unsafe paths fail closed when poisoned: `_sync_notifications`, `_drain_tc_inbox`/`_drain_tc_inbox_for_hook`, `_heal_pending_tool_calls`, `_inject_notification_pair`, `_handle_request`, `_handle_tc_wake`, and the pre-sleep heal all return early (and request refresh) instead of touching the interface.
- `prompt.py` → `tools.py` (via `agent._refresh_tool_inventory_section()`).
- All submodules read agent attributes (`agent._config`, `agent._session`, `agent._chat`, `agent._notification_fp`, `agent._notification_live_holder`, etc.) — the agent object is the shared state.
- `messaging.py` imports `from ..message import _make_message, MSG_REQUEST, MSG_TC_WAKE` and `from ..i18n import t as _t`. (`MSG_TC_WAKE` is no longer posted from `messaging.py` itself; the kernel sync owns wake transitions.)
- `tools.py` reads the injected intrinsic modules off `agent._intrinsic_modules` (populated by `_wire_intrinsics` from `BaseAgent(intrinsics=...)`); imports `from ..llm import FunctionSchema`, `from ..tool_glossary import append_tool_glossary`.
- `turn.py` imports `from ..state import AgentState`, `from ..loop_guard import LoopGuard`, `from ..safety_limits import ACTIVE_TURN_TOOL_CALL_EMERGENCY_LIMIT`, and `from ..tool_executor import ToolExecutor`. `LoopGuard` now gets its total-call emergency fuse from kernel-owned `safety_limits.py`, not from `manifest.max_turns` / `AgentConfig.max_turns`; `turn.py` records each tool-call batch before dispatch so dict-shaped results can expose the post-batch ACTIVE-turn progress counter.
- `lifecycle.py` imports `from ..state import AgentState`, `from ..token_ledger import sum_token_ledger`.

## State

- `_event_journal` is optional injected process-local ownership. Raw Core instances may leave it `None`; storage state belongs to the selected outer adapter (`base_agent/__init__.py:314`, `base_agent/__init__.py:346-347`).
- `identity.py` mutates `.agent.json` (manifest) and `system/system.md` (identity prompt section) via `_build_manifest` (`base_agent/identity.py:60-99`) and `_update_identity` (`base_agent/identity.py:32-49`). The manifest now includes a sanitized `llm` block (`provider`, `model`, `base_url`) read from the live `LLMService` via `_safe_llm_from_service` (`base_agent/identity.py:102-139`). The wrapper subclass extends with `preset` (active/default/allowed) read from `init.json` and re-applies the LLM safelist for defense-in-depth — see `src/lingtai/agent.py`. The identity prose renderer (`base_agent/__init__.py:139-181` inside `_build_identity_section`) folds both blocks into the cached system-prompt prefix only when present, so bare BaseAgent instances stay quiet.
- `lifecycle.py` writes `.agent.heartbeat` (`base_agent/lifecycle.py:207-211`; the file's lifetime equals the Python process's lifetime — see "Heartbeat-file lifetime contract" above), consumes signal files (`.interrupt`, `.refresh`, `.suspend`, `.sleep`, `.prompt`, `.clear`, `.inquiry`, `.rules`), and reads `.notification/*.json` via `_sync_notifications` on every tick.
- `turn.py` writes `history/chat_history.jsonl` via `_save_chat_history` (delegated to `__init__.py`), writes `.status.json` and `logs/token_ledger.jsonl`, and mutates the live notification holder on dict-shaped tool results by adding/removing the canonical `_meta.notifications` + `_meta.notification_guidance` payload. It also constructs the per-turn `ToolExecutor` with `working_dir=agent._working_dir` via the shared `_make_tool_executor` helper (`base_agent/turn.py:1225`), called from the request path (`base_agent/turn.py:986`) and the tc-wake path (`base_agent/turn.py:1071`), so the executor can spill oversized tool results into `<workdir>/tmp/tool-results/` — see `tool_result_artifacts.py:spill_oversized_result`. Notification-pair injection in `__init__.py` bypasses `ToolExecutor` and therefore the preventive cap, by design. **Retroactive compaction (issue #144)**: before either AED retry branch, `_compact_history_before_retry` walks `_session.chat.interface._entries` and rewrites already-committed `ToolResultBlock.content` larger than the 5K retroactive cap into the same `lingtai_tool_result_spill` manifest shape — entries / ids / pairing / `synthesized` flag are preserved. The helper logs a single bounded summary event `aed_history_compacted` (`scanned_blocks`, `compacted_blocks`, `original_chars_total`, `replacement_chars_total`, up to 16 `artifact_paths`) and, when at least one block was rewritten, calls `_save_chat_history(ledger_source="retroactive_compaction")` so the persisted history matches the wire before the rebuilt session replays it. **Over-window routing**: errors whose `str(exc)` matches `_OVER_WINDOW_MSG_FRAGMENTS` (context-window / prompt-too-long / token-count phrasing across providers) short-circuit the transient retry loop and take the deterministic branch with `source="aed_over_window"`, since retrying on the same wire would refire the same error. **WorkerStillRunningError is exempt** — that branch puts the agent ASLEEP without touching ChatInterface, so compaction never runs (it would race the still-alive worker future). The helper itself is exception-safe; any failure logs `tool_result_compaction_failed` and never re-raises. A hard pre-send gate (run compaction whenever the wire exceeds a threshold *before* attempting send) is left as a TODO in `_run_loop`.
- Notification persistence is owned by the injected Store. `messaging.py` atomically updates `system`; email and other producers publish through strict agent-bound Core helpers.
- `worker_recovery.py` writes `history/unfinished_turns/worker_still_running_*.json` (bounded/redacted unfinished-turn artifacts; atomic tmp+rename) and marks `prompt_injected_at` on the first safe request after restart. The poison flag and related `_llm_worker_*` fields are process-local in-memory state only (never persisted). `lifecycle._start` reads open artifacts via `rehydrate_worker_hang_recovery` to re-publish the high-priority notification after relaunch.
- `_inject_notification_pair()` still emits compact `notification_pair_injected` metadata for session summaries, while `_log_notification_block_injected()` records the durable actual-block history (`mode`, `call_id`, `sources`, full `_meta` envelope with `tool_meta`, `agent_meta`, `guidance`, `notifications`, and the compact `notification_guidance` ref/hook) for TUI `/notification` and log readers (`__init__.py:1441`, `__init__.py:1476`).

## Notes

- **LICC notification contract:** changes to notification sync, live-holder movement, `_meta.notifications`, `_meta.notification_persistent`, persistent notification delivery state, or producer authority/dismissal boundaries must check `src/lingtai/services/LICC_NOTIFICATION_CONTRACT.md` in the same change.
- **`__init__.py` is large.** This is intentional — the constructor, properties, state machine, hooks, cross-cutting infrastructure (`_save_chat_history`, `_log`, notification sync, legacy tc_inbox drain), and pass-through stubs all live here. The package is not "thin shell + 6 leaves"; the remaining code is genuinely cross-cutting or bound to the class definition.
- **Pass-through pattern.** Each extracted method becomes a 2-line stub in `__init__.py` (lazy import + call). This preserves the `BaseAgent` class interface while the implementation lives in submodules. The `self` → `agent` conversion is mechanical but `_heartbeat_loop` (~264 lines, 15+ cross-cluster calls) deserves extra-careful review.
- **Subclass overrides stay on `__init__.py`.** `_activate_preset`, `_activate_default_preset`, `_build_launch_cmd`, `_pre_request`, `_post_request`, `_on_tool_result_hook`, `_cpr_agent` are all overridden by the `Agent` subclass in the wrapper package. They must remain as methods on `BaseAgent`.
- **Telegram Task Card hooks (turn-local, route B).** Hooks in `__init__.py` implement the batched Task Card lifecycle — a rolling window of the newest **N ordinary tool rows** (env-configurable, default 1), not a single transient current step. `_setup_telegram_task_card` (`base_agent/__init__.py:2870-2937`) captures the current Telegram inbound route (`data.previews[0].message_ref` → `account:chat_id`), resolves the Telegram MCP client from `_mcp_clients_by_tool`, and stores a fingerprint-deduped turn-local `_telegram_task_card_context` (rows / generation / monotonic `clock` / `wall_clock` / heartbeat). `_on_tool_pre_dispatch_hook` (`base_agent/__init__.py:2342-2410`) appends one row per tool call in actual pre-dispatch order, opens a new epoch when no tool row is active (bumps `generation` so a stale heartbeat can't overwrite the window; supersedes a lingering API-error row, since a fresh tool batch means the LLM responded), then caps ordinary tool rows in place via `_cap_task_card_tool_rows` (`base_agent/__init__.py:2412-2428`) to `_task_card_max_tool_rows()` (`base_agent/__init__.py:83`, reads `LINGTAI_TASK_CARD_MAX_TOOL_ROWS`; positive int N, else fail-safe to 1; N is not otherwise clamped, so a very large N can exceed the render budget/transport limit) — the API-error row is retained and never evicted by the tool cap. `_on_tool_result_hook` (`base_agent/__init__.py:2279`) → `_freeze_task_card_row` (`base_agent/__init__.py:2309`) freezes the matching row (final whole-second elapsed + done) while others keep ticking; a row already scrolled out of the window has no matching `call_id`, so its late result is a no-op and cannot mutate the window. Rendering (create lazily, else edit the same card) is `_render_task_card` (`base_agent/__init__.py:2443`); the 0.5s heartbeat `_task_card_heartbeat_tick` (`base_agent/__init__.py:2724`) / `_start_task_card_heartbeat` (`base_agent/__init__.py:2759`) refreshes only active tool-row elapsed (floored to whole seconds) under the lock and no-ops on a stale `generation`. Each render reverse-calls the **private, unlisted MCP tool name** `_TASK_CARD_TOOL` (`= "_lingtai_telegram_task_card"`, `base_agent/__init__.py:71`) via `call_tool(_TASK_CARD_TOOL, {"sub_action": ..., ...})` — it sends **no** `action`; the Telegram server forces `_task_card_update` server-side (see `tools/mcp/ANATOMY.md`). A recursion guard skips the hook for `_TASK_CARD_TOOL` itself. Because `call_tool` reports tool-level failures as an error *dict* (and can also raise), the hooks inspect the result so an error is never treated as a created card (no fake success), compute the `message_id` once, stay fail-open, and make both error-dicts and raised exceptions observable via content-free warnings (phase/tool + status-or-exception-class only — never reasoning, chat id, account, card id, or provider text). LLM/provider API failures are surfaced onto the same card as one stable sentinel row by `_report_task_card_api_error` (`base_agent/__init__.py:2648`). `_teardown_telegram_task_card` (`base_agent/__init__.py:2939`) freezes the card on its concrete last batch (the retained newest-N rows + `✓` markers + final elapsed) with no generic overall `DONE` subject, surfaces a finalize error/exception the same content-free way, and clears context in `finally`.     Setup is called in `lifecycle.py:549` after `_sync_notifications()` inside the heartbeat loop.  Ordinary-request teardown runs in `turn.py:1123` (finally block) and in `_post_request` at `__init__.py:2277` (normal path).  tc-wake teardown runs in `turn.py:1332` (finally block around the wire-drive handler).  The real turn→ToolExecutor execute-hook wiring is at `turn.py:1732-1734` (`on_pre_dispatch_hook=getattr(agent, "_on_tool_pre_dispatch_hook", None)`).  The serial pre-future parallel hook block is at `tool_executor.py:1424-1432` (Phase 2, before `ThreadPoolExecutor` spawns futures), so parallel pre-dispatch order is deterministic.  The MCP consumer side is `TelegramManager._handle_task_card_update` (`src/lingtai/mcp_servers/telegram/manager.py:1374`), a private action not in SCHEMA, dispatched via `manager.handle()` at `src/lingtai/mcp_servers/telegram/manager.py:466` → `action == "_task_card_update"`; its update-first resident-card singleton and single card-level bare `HH:MM:SS UTC±HH` time line (no label, Jason #7213/#7216) are documented in `src/lingtai/mcp_servers/ANATOMY.md`.
- **Programmable Task Card controller (public `task_card` tool, Jason #7258/#7259).** `src/lingtai/kernel/task_card_controller.py` is the public, model-facing controller for the resident card's *programmable* slot (the automatic slot is route B above; both compose into one message — see `mcp_servers/ANATOMY.md`). Actions `start`/`inspect`/`retry`/`stop`: `start` validates a Python renderer file contained under the agent workdir (`_validate_renderer_path`, symlink-resolved `relative_to` containment), runs it once synchronously with the runtime interpreter under a timeout (`_run_renderer` → `subprocess.run([sys.executable, path])`), requires stdout to be **exactly one** schema-valid Task Card JSON object (`_validate_frame`; nonzero exit / timeout / non-object / multi-object / bad field type are synchronous tool errors and NO watch is created), then spawns a daemon watcher thread that re-runs on `interval_s` and projects each frame onto the private reverse channel `_TASK_CARD_TOOL` with `channel="programmable"`. After-handle failures mark the last-valid frame and emit a **deduped** structured LICC wake via `_enqueue_system_notification(source="task_card.error", …, skip_if_idempotency_key_exists=True)` — one wake per distinct error code, plus one `recovered` wake on the next good frame; raw renderer output/secrets never enter the wake. Registration is composition-root, not an intrinsic: `Agent._maybe_setup_task_card_controller` (`agent.py`) calls the module's `setup(agent)` (kernel module, `glossary_package=None`) once a Telegram MCP client is present, from both `connect_mcp`/`connect_mcp_http` (idempotent). `lifecycle._stop` calls `shutdown_for_agent_stop` to join watcher threads without any filesystem deletion.
