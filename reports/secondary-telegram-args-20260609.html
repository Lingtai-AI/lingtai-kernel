Secondary Telegram args validation

Repo: Lingtai-AI/lingtai-kernel · Branch: fix/secondary-telegram-args · 2026-06-09

结论：这个 PR 针对调查中发现的高频 Telegram secondary 参数畸形问题，在 kernel secondary 层做本地 normalize/validate。当前 origin/main 已经把 secondary 改为 read-only，所以本 PR 不重新允许 nested send/reply，只修 read 路径的目标字段和类型。

背景

dev-1/dev-3 日志里出现大量 secondary Telegram 失败，例如 chat_id:"" 或 chat_id:"6859932159" 直接进入 Telegram MCP，最终由 MCP schema 报 '' is not of type 'integer' / '6859932159' is not of type 'integer'。

根因是 secondary 的 provider-facing schema 是跨工具共享的轻量 schema，runtime 只检查 tool/action/递归，不按目标工具/action 做必填字段和类型校验。

变更

文件变更
src/lingtai_kernel/secondary_tools.py新增 SECONDARY_READ_TARGET_FIELDS 和 normalize_secondary_args()：剔除空字符串与 limit=0 占位；校验 read 目标字段；Telegram digit-string chat_id 转 int；非整数 Telegram chat_id 本地拒绝。
src/lingtai_kernel/tool_executor.py_validate_secondary() 在 action/递归检查后调用 normalize/validate；失败时返回本地 _secondary.status=error，不 dispatch 到 MCP。
tests/test_tool_executor.py新增回归测试：Telegram digit-string chat_id normalize；空 chat_id 本地拒绝；email.read 空 email_id 本地拒绝。

行为示例

{"action":"read", "chat_id":"6859932159", "limit":0}
→ {"action":"read", "chat_id":6859932159}

{"action":"read", "chat_id":""}
→ _secondary error: secondary telegram.read requires chat_id

{"action":"read", "email_id":""}
→ _secondary error: secondary email.read requires email_id

风险与边界

  • 不影响 top-level Telegram send/reply/read。
  • 不重新开放 secondary send/reply；current main 的 read-only policy 保持不变。
  • 对 Telegram 数字字符串 chat_id 做宽容转换；对空目标字段做本地错误，避免 MCP noise。
  • Feishu/WhatsApp chat_id 仅要求存在；没有强制 int，因为这些 channel 的 ID 格式可能不是 Telegram 风格。
