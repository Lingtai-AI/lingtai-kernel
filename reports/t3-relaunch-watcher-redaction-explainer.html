T3 Explainer — Redact secrets in relaunch-watcher event persistence

Branch: reliability/t3-secret-redaction-20260624. Wave 1 Track T3 (secret redaction / log security).

Conclusion

The refresh/relaunch watcher subprocess persisted its own events to logs/events.jsonl without redaction, so secret-shaped values could leak into the durable event log. This PR routes that watcher's inline writer through the existing kernel redactor so stderr_tail, cmdline, and error previews are scrubbed before persistence.

Why PR #315 did not already cover this

Open PR #315 applies lingtai_kernel.trace_redaction.redact_text to the Telegram conversation preview boundaries in src/lingtai/mcp_servers/telegram/manager.py. That is a different surface. It does not touch the kernel's refresh/relaunch path.

The in-process logging service (services/logging.py → CompositeLoggingService.log) already wraps every event in redact_for_trajectory before the JSONL/SQLite write. The gap is that the relaunch watcher runs in a separate subprocess and writes events.jsonl through its own inline log(), which never reached that redaction step.

The leak path

base_agent/lifecycle.py generates a self-contained Python watcher script (run via python -c) that supervises relaunch. Its inline log() appends raw JSON lines to events.jsonl. Three fields carry untrusted text:

FieldSourceLeak risk
stderr_taillast 500 chars of the relaunched process's stderrA boot/config traceback can echo an env var or token (e.g. a bot token in a config error).
cmdlineps -o command= of a duplicate processCommand line may include a token passed as a CLI argument.
errorstr(exc) for spawn/kill failuresException text may quote credential-bearing values.

Because these went straight to disk, the durable event log could retain a raw secret even when the live conversation never reprinted it — the exact persistence hazard trace_redaction exists to prevent.

The fix

The watcher's single write chokepoint is its inline log(). We add a redaction step there, mirroring CompositeLoggingService.log's redact_for_trajectory(event):

try:
    from lingtai_kernel.trace_redaction import redact_text as _redact_text
except Exception:
    def _redact_text(s):
        return s
def _redact_entry(value):
    if isinstance(value, str):
        try:
            return _redact_text(value)
        except Exception:
            return value
    if isinstance(value, dict):
        return {k: _redact_entry(v) for k, v in value.items()}
    if isinstance(value, (list, tuple)):
        return [_redact_entry(v) for v in value]
    return value
def log(typ, **kw):
    entry = {'type': typ, 'address': addr, 'agent_name': name, 'ts': time.time(), **kw}
    entry = _redact_entry(entry)
    ...write json...
  • Single source of truth. The watcher imports the same redact_text the in-process path uses; it does not duplicate the regex set.
  • Fail-open. If the import or a redaction call ever fails, log() degrades to identity rather than crashing the relaunch supervisor — redaction must never break recovery.
  • Recursive over the whole entry. Every string field is scrubbed, so future fields are covered without revisiting each call site.

Tests (all green)

Added to tests/test_perform_refresh_handshake.py. All token literals are fake, fixed-shape values used only to exercise the redaction regexes:

  • test_refresh_watcher_script_embeds_redactor — the generated script wires in trace_redaction/redact_text at the write chokepoint.
  • test_refresh_watcher_log_redacts_secret_fields — executing the generated log() with a fake-token-bearing stderr_tail and cmdline writes a record where no fake token survives and a <REDACTED:*> placeholder appears; records remain valid JSON with type/attempt/pid intact.

Fake shapes covered: Telegram bot token (123456789:AAA…), Bearer …, sk-… API key, and a keyed env assignment (BOT_TOKEN=…).
