DaemonGroup v0: daemon-authored peer_send

Prepared 2026-06-14 for the LingTai kernel PR from feat/daemon-group-v0-peer-send.

Conclusion

Ready for PR review. This feature adds explicit parent-owned DaemonGroups where opt-in LingTai, Codex, and Claude Code daemons can send bounded peer messages to sibling daemons through one parent-mediated policy and delivery core.

The prior blocker — Codex resume failing with no rollout found — was real. The root cause was a race: Codex emits thread.started.thread_id before the initial rollout is fully resumable. The fix returns busy until the initial Codex future is done, then resumes normally.

What changed

  • Added src/lingtai/core/daemon/peer.py: pure dataclasses and helpers for group roster, peer message envelopes, strict CLI sentinel parsing, policy authorization, provenance banners, roster notices, and author contracts.
  • Extended DaemonManager with group_create, group_status, and group_reclaim.
  • Added opt-in peer_author support for LingTai in-process daemons, Codex CLI daemons, and Claude Code CLI daemons.
  • Added native in-process peer_send tool for LingTai authors; parent remains the only router.
  • Added strict post-turn CLI sentinel parsing for Codex/Claude Code authors: one block only, no natural-language parser, daemon-authored identity fields rejected.
  • Added body-free peer routing traces into the parent log and CLI author's run dir so daemon(check) can explain send outcomes without leaking message text.
  • Added Codex resume guard: do not spawn codex exec resume while the initial Codex future is still running.
  • Updated src/lingtai/core/daemon/ANATOMY.md for the new peer module, group APIs, and routing flow.

Safety model

BoundaryBehavior
Parent-only routingDaemons never get the full daemon tool and cannot directly address process internals. All peer sends pass through the parent.
Explicit groupsOnly members in group_create can send/receive. Groups snapshot stable run_ids, not reusable handles alone.
Call-time revocationgroup_reclaim revokes by policy check; no dynamic tool-schema mutation is required.
No hidden queuesent means accepted into the target follow-up path. Busy/done/not-ready are terminal outcomes for v0.
Metadata-only logsEvents record handles, statuses, ids, byte counts, and reasons, but not peer message bodies.

Known non-goals / follow-ups

  • No durable outbox/scanner/ack/dedupe in v0.
  • No broadcast or fanout.
  • No multi-group/cross-group routing.
  • No free-form natural-language parser; CLI authors must emit the strict sentinel.
  • OpenCode/Cursor are receive-only for v0 unless a future author contract is designed for them.
