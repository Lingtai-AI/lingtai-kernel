Issue #193: stop over-budget turns before provider dispatch

This PR adds a local hard context ceiling in SessionManager.send() so LingTai recovers or blocks before making an LLM API call that is already estimated to exceed the configured context budget.

Context Why this matters

Issue #193 reported that the runtime could show extreme context pressure but still try to send another request. Existing warning and provider-overflow recovery paths were reactive: they either warned the agent or handled a provider rejection after an oversized request was attempted.

The safer behavior is to perform a deterministic pre-send estimate of the canonical wire plus the pending message, then either reduce local history or stop locally before provider dispatch.

Root cause Warning-only pressure handling

Before this change, request handling could publish context pressure warnings and still proceed into SessionManager.send(...). Tool-loop continuations and wire-drive sends likewise had no central hard gate that accounted for pending text/tool-result payloads before the API call boundary.

What changed Implementation summary

Central hard gate

  • Adds AgentConfig.context_hard_pressure, defaulting to 0.98.
  • Checks the gate in SessionManager.send() after prompt/tool refresh and health checks, but before llm_call and provider dispatch.
  • Estimates current canonical wire plus pending str, None, or completed ToolResultBlock list messages.

Bounded local recovery

  • Runs one deterministic oversized-tool-result compaction attempt through an injected callback.
  • For replayable user text only, may force a system-authored context_forget(source="hard_ceiling") and retry the estimate.
  • If still over budget, returns a local terminal assistant response without calling the provider.

Tool-result safety on local block

If a post-tool continuation is blocked, the completed tool results are committed locally first. This preserves side effects already performed by tools and prevents the agent from blindly re-running them on the next turn.

Server-side session consistency

After recording a local hard-gate assistant note, the session is rebuilt from canonical history. The OpenAI Responses server-side adapter now seeds its next request from canonical history when there is no previous_response_id chain, including any pre-request hook entries drained before that first post-rebuild request.

Follow-up: server-side OpenAI Responses transcript

After review, the PR was updated so server-side OpenAI Responses sessions also maintain the local canonical transcript while continuing to use previous_response_id for provider state. This makes the hard-gate estimate include turns hidden behind the provider response chain and ensures a rebuild after a blocked tool-result continuation seeds a valid assistant function-call / tool-result pair.

Non-goals What this PR intentionally does not do

  • It does not change the human/agent warning ladder UX beyond adding the pre-send hard stop.
  • It does not replace provider-specific overflow handling; provider rejections can still occur for estimator mismatch or provider-side accounting differences.
  • It does not add new wake notifications on hard-gate block; the block is returned as local model-visible assistant text in the active turn.
  • It does not broaden context compaction beyond the existing deterministic oversized-tool-result artifact path.
