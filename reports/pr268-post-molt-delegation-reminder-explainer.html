Issue #256 — Post-molt delegation reminder

Prepared for LingTai kernel PR #268. Branch: fix/post-molt-delegation-reminder. Rebased for conflict resolution onto canonical main 68e1d00; conflict-fix daemon verified branch state at a9eab25 before this report-only refresh.

TL;DR

This change adds a separate post-child-delegation notification after both agent-initiated and system-forced molts. It reminds the post-molt agent that direct delegated work may still exist, without blocking molt and without taking lifecycle action automatically.

PR-ready locally: an independent Codex daemon review found one real bounded-scan bug, patched it, and a later conflict-resolution daemon rebased the branch onto current canonical Lingtai-AI/lingtai-kernel main with focused validation passing.

Baseline

Before this patch, molt already had a post-molt continuation reminder via .notification/post-molt.json, and normal notification sync could wake/surface it after context shedding. But molt did not create a structured reminder for delegated work that might still be alive after the parent’s conversation context was replaced.

Issue #256’s clean design target is not to refuse or delay molt. The safer behavior is: finish the molt, then surface a bounded awareness snapshot so the new session can choose whether and how to inspect children or daemons.

What was done

1. New notification channel

src/lingtai_kernel/notifications.py now allowlists post-child-delegation, keeping this reminder distinct from both the pressure-warning molt channel and the existing post-molt continuation channel.

2. Awareness-only post-molt publisher

src/lingtai_kernel/intrinsics/psyche/_molt.py adds _publish_post_child_delegation_reminder(), called immediately after _publish_post_molt() in:

  • agent-initiated _context_molt(...)
  • system-forced context_forget(...)

The payload includes:

schema_version source_agent initiator source molt_count awareness_only: true automatic_lifecycle_actions: false descendant_scan: "direct_only" counts limits

3. Bounded direct-child scan

Avatar scan reads only the parent’s delegates/ledger.jsonl and each direct child’s .agent.json / .agent.heartbeat. It does not recurse into child ledgers and does not read child history, mailbox, notifications, knowledge, or other private stores.

Daemon scan reads only parent-side daemons/*/daemon.json and .heartbeat. It does not read daemon transcripts, events, history, result files, or outputs beyond the bounded fields already present in daemon.json.

4. Tail-window fix from final review

The first implementation read the first 1000 avatar ledger lines. Because delegates/ledger.jsonl is append-only, that could omit newer direct children for a long-lived parent. The final review daemon patched this: _read_avatar_ledger_tail() now reads the newest bounded window, walks it newest-first, and keeps the freshest record per child. A regression test covers a ledger with 1000 old records followed by a recent live child.

Risks and decisions

  • Direct-only is intentional. The payload explicitly says descendant_scan: "direct_only". Recursive descendant scanning can be added later with separate depth/node/truncation semantics.
  • Counts are bounded-window counts. If a parent has more than 1000 avatar ledger records, older children outside the newest window may be omitted. limits.avatars_truncated / limits.truncated make this visible.
  • No automatic lifecycle action. The notification tells the agent not to CPR, interrupt, reclaim, suspend, or message children automatically.
  • Privacy boundary is narrow. The implementation avoids child private stores and daemon detailed artifacts.
