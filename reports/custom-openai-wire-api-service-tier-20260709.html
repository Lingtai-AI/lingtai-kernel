OpenAI-Compatible Wire Controls

Local kernel PR prep · branch fix/custom-openai-wire-api-service-tier · commit b8a878a9 · base origin/main@2a376f5b

TL;DR

Conclusion: this local commit adds manifest-level control for OpenAI-compatible providers that need explicit Responses API routing or request-level service tier passthrough. It is aimed at Codex subscription access routed through sub2api/intermediate OpenAI-compatible providers, where endpoints may be Responses-only or expose fast-tier behavior through OpenAI-compatible fields.

Baseline

  • Before: api_compat selected protocol family, but custom OpenAI-compatible providers did not have a first-class manifest field to force responses vs chat_completions.
  • Before: service-tier passthrough was not consistently propagated from manifest/provider defaults into OpenAI-compatible request kwargs.
  • Repo state used for prep: clean worktree from origin/main@2a376f5b; original dirty branch was preserved as <agent-artifacts>/lingtai-custom-api-pr-prep/kernel-custom-api-20260709.patch.

What Was Done

Manifest/provider defaults

manifest.llm.wire_api now accepts auto, chat_completions, or responses. service_tier is passed through as an optional string.

Adapter routing

OpenAI and custom OpenAI-compatible factories map wire_api="responses" to forced Responses mode and wire_api="chat_completions" to Chat Completions mode.

Diagnostics

Wrong-shaped Chat Completions responses now fail with an explicit hint that the endpoint may require wire_api="responses".

Anatomy

LLM anatomy files were updated to mention the new manifest/provider-default fields and OpenAI/custom adapter surfaces.

Risks / Decisions

  • Alias precedence: backward-compatible use_responses* aliases can still interact with wire_api. The code keeps aliases for compatibility and tests the preferred wire_api path.
  • Codex service tier: service_tier is passed into the Codex Responses path. If an upstream-compatible backend rejects the field, users can omit/default it.
  • No remote issue linked yet: dev-guide prefers issue → branch → PR. Filing/linking an issue requires explicit authorization, so the PR body draft uses TBD.
  • Decision: api_compat remains the protocol family; wire_api is only the OpenAI-family wire selector.
